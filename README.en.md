## Project Description

CodeFlow is a comprehensive workflow engine that automates your AI-powered project development processes. Managed through its core files, `codeflow.xml` (the workflow engine) and `prompt.xml` (the workflow configuration and guidance), CodeFlow significantly enhances code quality and maintainability through advanced pattern management, architectural governance, integration excellence, roadmap intelligence, and error management mechanisms. All project metrics, reports, temporary files, and other critical outputs generated by the system are stored in a structured manner under the `.project_meta/` directory.

**Why CodeFlow?**

* **Consistency:** Standardizes code quality and architectural integrity across the team
* **Efficiency:** Automates routine tasks and accelerates development processes
* **Predictability:** Improves project planning and risk assessment
* **Scalability:** Supports growing codebases and teams
* **Sustainability:** Enables proactive management of technical debt

CodeFlow operates with two complementary core XML files that provide a comprehensive framework for automating and optimizing the project development process:
*   **`codeflow.xml`**: Defines the main workflows, their steps, the tools to be used, and the sequential logic of these steps. It is the primary operational engine of the system.
*   **`prompt.xml`**: A configuration and guidance file that details how the workflows in `codeflow.xml` should operate for a specific project or context, defining objectives, quality standards, expected outputs, and human interaction points. It is used together with `codeflow.xml` in tasks given to the LLM to customize the behavior of the process.

### Main Components

CodeFlow's core capabilities are delivered by the following integrated main components and systems:

* **Workflow Engine (`codeflow.xml`)**: Defines all steps, conditions, and actions in the project lifecycle. Performs all necessary initial project setup (e.g., creating metric and report directories under `.project_meta/`) through steps like `initialize_project`.
* **Workflow Configuration (`prompt.xml`)**: Defines project-specific inputs such as quality expectations, targeted workflow steps, custom metrics, reporting formats, and situations requiring human intervention.
* **Pattern Management Engine ( `learn_patterns` step and related tools)**: Automatically detects, classifies, develops, tracks the evolution of code patterns, and manages their metrics, history, and relationships under `.project_meta/.patterns/`.
* **Architecture Validation System (`analyze_initial_architecture` step and tools like `architecture_analyzer`)**: Ensures the codebase adheres to architectural principles and generates relevant metrics and reports under `.project_meta/.architecture/`.
* **Integration Orchestrator (`integration_phase` step and related tools)**: Manages multi-level integration tests, performs compatibility analyses, and stores metrics and reports under `.project_meta/.integration/`.
* **Roadmap and Story Manager (`create_roadmap_and_stories` step and tools like `roadmap_manager`)**: Optimizes project planning, breaks down requirements into stories, manages dependencies, and maintains metrics under `.project_meta/.stories/`.
* **Advanced Error Analysis and Management System (`handle_error` step and tools like `error_analyzer`)**: Performs multi-dimensional error classification, root cause analysis, offers intelligent recovery strategies, and houses metrics and reports under `.project_meta/.errors/`.
* **Dynamic Documentation System (various documentation tools)**: Generates and validates continuously updated documentation about the code and project, managing its metrics, validation results, and visualizations under `.project_meta/.docs/`.

These systems work in concert to provide quality, consistency, and efficiency in modern software development processes.

## Key Features

### 🔄 Integrated and Configurable Workflow

CodeFlow offers a comprehensive workflow engine that automates project development based on a validated Product Requirements Document (PRD) and project objectives defined in `prompt.xml`. While `codeflow.xml` defines the main process, `prompt.xml` specifies how this process should operate according to specific needs (e.g., which steps to focus on, quality standards, expected outputs).

* **Automatic PRD validation and analysis (`validate_prd` step)**
* **AI-powered architectural design and evaluation guided by `prompt.xml` (`analyze_initial_architecture` step).** Outputs are stored under `.project_meta/.architecture/`.
* **Smart modular structure and dependency management (`create_roadmap_and_stories` step).** Outputs are stored under `.project_meta/.stories/`.
* **Iterative and cyclical development process automation.**
* **Phased integration and test automation (`integration_phase` step).** Outputs are stored under `.project_meta/.integration/`.
* **Error management and knowledge accumulation (`handle_error` step).** Outputs are stored under `.project_meta/.errors/`.
* **Seamless feedback and continuous improvement.**
* **Dynamic, real-time documentation system (documentation tools).** Outputs are stored under `.project_meta/.docs/`.
* **Project initialization setup (`initialize_project` step):** Creates all necessary metric, report, and working directories under `.project_meta/`.

```xml
<!-- Example Workflow Step (from codeflow.xml 'analyze_initial_architecture') -->
<step id="analyze_initial_architecture">
  <description>Establish comprehensive, robust initial architecture based on the validated PRD and prompt.xml specifications through multi-dimensional analysis. Relevant metrics and review files will be generated and validated under .project_meta/.architecture/.</description>
  <condition>After PRD is ready and validated, and project is initialized.</condition>
  <tools_setup>
    <tool name="architecture_analyzer"/>
    <tool name="prd_analyzer"/>
    <tool name="code_reviewer"/>
    <tool name="reporting_tool"/>
  </tools_setup>
  <action>
    <phase name="Architectural Analysis and Design">
      Use `architecture_analyzer` to perform comprehensive architecture analysis based on PRD and `prompt.xml` quality/performance targets:
        - Evaluate multiple architecture styles/patterns against PRD requirements.
        - Perform quantitative analysis of quality attributes (e.g., performance, scalability, security, maintainability).
        - Generate architecture quality scores for each candidate approach.
        - Identify potential risks and trade-offs for each option.
      Use `prd_analyzer` to ensure deep understanding of functional and non-functional requirements influencing architecture.
    </phase>
    <phase name="Documentation and Metric Generation">
      The `architecture_analyzer` and `reporting_tool` will generate and store detailed architectural documents, decision logs, and all relevant metrics (including those specified in `prompt.xml`) in `.project_meta/.architecture/architecture_metrics/` and `.project_meta/.architecture/reviews/`.
      All generated files will be validated for completeness and correctness.
    </phase>
  </action>
  <output>
    <item type="report" path=".project_meta/.architecture/main_architecture_document.json" description="Main architecture document detailing the chosen approach, rationale, and design."/>
    <item type="metrics" path=".project_meta/.architecture/architecture_metrics/" description="Directory containing all generated architectural metrics."/>
    <item type="reviews" path=".project_meta/.architecture/reviews/" description="Directory containing architectural review files and feedback."/>
    <item type="status" description="Architecture analysis complete. All metrics and reports are generated and validated in their respective .project_meta/architecture/ subdirectories."/>
  </output>
  <error_management>
    <handle error_type="prd_ambiguity" action="request_clarification_from_user"/>
    <handle error_type="tool_failure" action="retry_with_alternative_tool_or_log_error"/>
    <handle error_type="metric_generation_failure" action="log_error_and_proceed_with_available_data"/>
  </error_management>
</step>
```

## Installation and Usage

### Prerequisites

-   Access to a Large Language Model (LLM) (e.g., GPT-4, Claude, Llama, etc.). The LLM must be capable of processing `codeflow.xml` and `prompt.xml` files in XML format and understanding the instructions.
-   The `codeflow.xml` (workflow engine) and `prompt.xml` (project-specific workflow configuration) files.
-   Project code to be analyzed or developed, and related documents (e.g., PRD).

### Installation

CodeFlow is not installed like a traditional software package. The `codeflow.xml` and `prompt.xml` files are provided directly to the LLM, either as a system message (system prompt) or as part of the user message:

1.  Obtain the latest versions of `codeflow.xml` and `prompt.xml` files from the CodeFlow repository (or as provided to you).
2.  Copy the content of these XML files.
3.  In your chosen LLM platform, provide this content along with your instructions to the LLM. Typically, the `codeflow.xml` and `prompt.xml` contents are placed before your actual query or in a dedicated system message field.
    *   **Example:** "Using the `codeflow.xml` and `prompt.xml` content below, analyze the provided PRD document and create an initial architectural plan. Save the outputs under the `.project_meta/` directory structure as specified.
        
        `<codeflow_xml_content_goes_here>`
        
        `<prompt_xml_content_goes_here>`
        
        `<PRD_content_or_reference_goes_here>`"
4.  The LLM will interpret these XML files to understand the defined workflow and configuration, then execute your instructions within that framework.

CodeFlow stores all its outputs and working files under a folder named `.project_meta/`, which it will create in the root directory of your project. Therefore, ensure the LLM has the necessary permissions to read from and write to this directory (if the LLM interacts with a local file system).

### Basic Usage Scenarios

Instructions given to the LLM usually include the content of `codeflow.xml`, `prompt.xml` (optionally for targeted use), and a specific task.

```plaintext
# Initialize the project and analyze PRD (uses default settings from prompt.xml)
"[codeflow.xml] [prompt.xml] Analyze this PRD and initialize the project. Save reports under .project_meta/architecture/.
[PRD Content]"

# Focus on a specific step (e.g., learning patterns)
"[codeflow.xml] [prompt.xml] Learn patterns from this codebase and save them under .project_meta/patterns/.
[Code Samples or Codebase Reference]"

# Get a general report on the current project status and metrics
"[codeflow.xml] [prompt.xml] Analyze the current project status and generate a summary report from metrics under .project_meta/."

# Query a specific set of metrics
"[codeflow.xml] [prompt.xml] Report the integration metrics (.project_meta/integration/metrics/) and error trends (.project_meta/errors/metrics/trend_analysis.json)."

# Request error analysis
"[codeflow.xml] [prompt.xml] Analyze potential errors in the provided log files and report them under .project_meta/errors/.
[Log File Content or Reference]"

# Check documentation quality and currency
"[codeflow.xml] [prompt.xml] Evaluate the project's documentation quality (.project_meta/docs/metrics/) and report deficiencies."

# Check the completion status and dependencies of a user story
"[codeflow.xml] [prompt.xml] Check the status, dependencies, and related metrics under .project_meta/stories/ for story 'story_id_xyz'."

# Get information about a human interaction point or ask what to do if triggered
"[codeflow.xml] [prompt.xml] What are the details for the human interaction point defined for the 'critical_anti_pattern_detected' event? What notifications will I receive if it's triggered?"

# Support for planning the next iteration
"[codeflow.xml] [prompt.xml] Prioritize stories from .project_meta/stories/product_backlog.json, analyze their dependencies, and provide recommendations for the next sprint."
```

## User Guide

Effectively using CodeFlow requires understanding how `codeflow.xml` workflow definitions and `prompt.xml` project-specific configurations work together.

### The `.project_meta/` Directory: Project Brain

All outputs, metrics, reports, logs, and temporary analysis files generated by CodeFlow are centrally collected under the `.project_meta/` directory. This directory contains detailed information about the project's current state, history, and evolution. Key subdirectories include:

*   `.project_meta/.architecture/`: Architectural documents, metrics, reviews, decision logs.
*   `.project_meta/.patterns/`: Identified patterns, their metrics, evolution, relationships, usage history.
*   `.project_meta/.stories/`: Product backlog, dependency maps, metrics, risk assessments, visualization data.
*   `.project_meta/.integration/`: Integration test reports, metrics, compatibility matrices, failure analyses, visualizations.
*   `.project_meta/.errors/`: Classified errors, root cause analyses, metrics, knowledge base, visualizations.
*   `.project_meta/.docs/`: Auto-generated documents, quality metrics, validation reports, reference analyses, visualizations.
*   `.project_meta/.logs/`: Detailed operational logs of workflow steps.
*   `.project_meta/.temp/`: Temporary analysis files and workspace.

This structure provides organized access to data related to all aspects of the project.

### Customizing the Process with `prompt.xml`

The `prompt.xml` file allows you to tailor the general workflow defined by `codeflow.xml` to the specific needs of your project. Some important sections of `prompt.xml` and their uses:

*   **`<target_workflow_steps>`**: Used to focus on specific `codeflow.xml` steps or define special emphases for these steps (e.g., security testing for `integration_phase`).
*   **`<quality_standards>`**: Specifies the quality metrics (e.g., code coverage, performance targets, response time) and threshold values that the project must adhere to. CodeFlow analyzes and reports against these standards.
*   **`<output_expectations>`**: Defines the format, content, and storage paths for outputs expected from workflow steps. For example, release notes, security reports, or custom metric aggregations.
*   **`<pattern_metrics>`, `<architecture_metrics>`, etc.**: Define specific metrics to be tracked and their targets. These metrics are stored in the respective `.project_meta/` subdirectories.
*   **`<human_interaction_points>`**: Defines situations where the automated process requires human intervention or approval. (See the "Human Interaction Points" section below for details.)

## Metrics and Analytics

## Advanced Configuration

CodeFlow's advanced configuration capabilities allow for customization and optimization of the workflow and its components.

### Human Interaction Points (`<human_interaction_points>`)

While CodeFlow aims to automate a large part of the development process, some situations may require human intuition, decision-making, or approval. The `<human_interaction_points>` section in `prompt.xml` is designed to manage such scenarios.

**How It Works**

Each human interaction point is defined by a specific `event` and an `action` that CodeFlow will take when this event occurs.

*   **`event`**: Represents a special condition or threshold that CodeFlow might encounter during its workflow. For example, "critical anti-pattern detected," "a security vulnerability found," or "major PRD deviation detected."
*   **`action`**: Defines what CodeFlow will do when the specified event occurs. This typically includes actions like "request user approval," "generate a detailed report and submit for human review," "notify a specific expert," or "pause the process and await human input."

**Example:**

An example from the `prompt.xml` file:

```xml
<human_interaction_points>
  <point id="hip_001">
    <event type="critical_anti_pattern_detected">
      <condition>Severity > 8 AND ImpactScope = "core_module"</condition>
      <message_template>Critical anti-pattern [pattern_name] detected in [file_path] with severity [severity_score]. This may impact core functionality. Please review and advise on remediation strategy.</message_template>
    </event>
    <action type="request_user_intervention">
      <notification_channel>email</notification_channel>
      <recipient>lead_developer@example.com</recipient>
      <escalation_rule after="24h" to="project_manager@example.com"/>
      <required_input>Approve | Reject | Suggest Alternative</required_input>
    </action>
  </point>
  <point id="hip_002">
    <event type="major_prd_deviation_detected">
      <condition>DeviationImpactScore > 0.75 AND DeviationType = "scope_creep"</condition>
      <message_template>Major PRD deviation (scope creep) detected. Estimated impact on timeline: [estimated_delay_weeks] weeks. Original requirement: [original_requirement_summary]. New finding: [new_finding_summary]. Please assess and confirm if development should proceed with this new scope or if PRD needs revision.</message_template>
    </event>
    <action type="pause_affected_tasks_and_request_decision">
      <notification_channel>slack</notification_channel>
      <recipient>#product_owners_channel</recipient>
      <required_input>Proceed | Revise PRD | Defer Feature</required_input>
    </action>
  </point>
  <!-- ... Other interaction points ... -->
</human_interaction_points>
```

In this example:

1.  `hip_001`: If a critical anti-pattern with severity > 8 and impact scope "core_module" is detected, CodeFlow sends an email notification to the lead developer, requesting advice on a remediation strategy. If there's no response within 24 hours, it escalates to the project manager.
2.  `hip_002`: If a major PRD deviation (scope creep) is detected, CodeFlow pauses affected tasks, sends a Slack notification to the product owners, and requests a decision on whether to proceed with the new scope or revise the PRD.

**Importance and Benefits:**

*   **Control and Flexibility:** Allows for human intervention at critical decision points where full automation might be risky or undesirable.
*   **Risk Management:** Enables early assessment of potential issues by human experts.
*   **Collaboration:** Establishes an effective bridge for collaboration between the automated system and the development team.
*   **Transparency:** Informs relevant stakeholders about significant events and decisions.

Human interaction points combine the power of CodeFlow with human intelligence and experience, leading to a more robust and reliable development process.

## Frequently Asked Questions

### General Questions

**Q: Which LLMs can be used with CodeFlow?**
A: CodeFlow can be used with advanced LLMs such as GPT-4, Claude, Llama. It is compatible with any LLM that can understand complex XML structures and instructions.

**Q: How can CodeFlow be integrated into an existing project?**
A: You can integrate CodeFlow XML files into existing projects by providing them as a system prompt to the LLM along with your codebase content.

**Q: Do I need to manually edit the XML files?**
A: Yes, you can edit the CodeFlow XML files according to your project's needs. You can manually customize the files following the XML schemas or get help from an LLM for this.

### Technical Questions

**Q: How scalable is CodeFlow?**
A: CodeFlow's scalability depends on the capacity of the LLM you are using. Modern LLMs can provide effective analysis even in large codebases, but for very large projects, you may need to split the code into parts.

**Q: How do I define custom patterns and metrics?**
A: You can add custom pattern definitions and metrics in the codeflow.xml and prompt.xml files. You can make new definitions according to the XML schema.

**Q: How is CodeFlow integrated with LLM APIs?**
A: You can integrate CodeFlow XML files into the system prompt section of your API calls. You can achieve integration using LLM services such as OpenAI API, Anthropic API, or Hugging Face API.
